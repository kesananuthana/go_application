//Go Routines
A Goroutine is a lightweight thread managed by Go.
It allows your program to run multiple functions at the same time (concurrently).
To run a function as a goroutine, just add go keyword
Normal function: Runs one after another
Goroutine: Runs independently (in parallel/concurrently)

why we use goroutines :
Because they make programs faster and efficient when tasks don’t depend on each other.
Without Goroutines:
One task finishes → then next starts
Total time = sum of all tasks
With Goroutines:
Multiple tasks run together
Total time = much less

where goroutines are used:
In APIs / Web Servers

When many users send requests:
->In APIs / Web Servers
Each request runs as a goroutine
So server handles many users at once
->File Processing
Reading multiple files
Writing logs
Data processing
->Network Calls / APIs
Calling multiple APIs
Fetching data from different services
->Background Tasks
Sending emails
Notifications
Logging
->Real-Time Systems
Chat apps
Streaming
Gaming

How it behaves:
Starts when you use the go keyword
Runs in the background
Does not block the main program
Managed by Go runtime (not directly by OS threads)

When we use go before a function, it runs that function concurrently in the background. 
The main function does not wait for it. If the main function finishes before the goroutine completes, the program exits and the goroutine may not finish executing. 
That’s why we use time.Sleep, WaitGroup, or channels to ensure goroutines complete properly.

=>Wait Group:
WaitGroup is used to: Make main() wait until all goroutines finish
ex:func main() {
	var wg sync.WaitGroup
	wg.Add(3) //it means we are starting 2 goroutines
	go task("A", &wg)
	go task("B", &wg)
	task("c", &wg)
	wg.Wait() //wait until finish
	fmt.Println("All tasks are completed")
}

func task(name string, wg *sync.WaitGroup) {
	defer wg.Done() // tell "I am finished"
	fmt.Println("I am ", name)
}
here why we used pointer(&wg). beacuse it uses the same address not copy. 
sync.WaitGroup : This is a type from Go:
A structure that helps wait for goroutines to finish
wg *sync.WaitGroup means we are passing a pointer so all goroutines share and modify the same WaitGroup instance instead of working on copies.
defer &wg.Done() : To ensure the goroutine ALWAYS signals completion safely
We pass &wg to share the same WaitGroup, and use defer wg.Done() to guarantee that the goroutine always marks itself as finished even if it exits early.
In above code I didn't give goroutine to  task c even also it works:
Task C works because WaitGroup depends only on Done() calls, not whether the function is a goroutine or a normal function.
Yes, WaitGroup works with both goroutines and normal functions, because it only tracks Done() calls, not how the function is executed.
=>WaitGroup does NOT track:
goroutines
threads
background tasks
it only tracks: how many times Done() is called

Channels:
A channel is a way for goroutines to: Send data to each other safely
Simple meaning
Goroutine A = sender 
Goroutine B = receiver 
Channel = pipe between them 
2. Why do we need channels?
Without channels:
Goroutines run independently
No safe way to share data
Risk of confusion or race conditions
Channels solve this by:
Safe communication + synchronization
Important:
Channel is blocking
sender waits if no receiver
receiver waits if no sender

Channels can work with normal functions only if send and receive don’t block each other.

There are 2 ways this can happen:
1. Using buffered channel
func main() {
	ch := make(chan string, 1) //It has space to store 1 value
	task(ch)
	msg := <-ch
	fmt.Println(msg)
}

func task(ch chan string) {
	ch <- "Normal function"
}

Why companies use channels
✔ Safe communication
✔ No data corruption
✔ Easy concurrency
✔ Clean design
✔ Better performance

Creating Http Servers
For creating web servers we need net/http packages
net/http is a standard library package in Go.
It helps you create web servers
It helps you handle HTTP requests & responses
net = networking
It helps programs talk to each other over a network
it handles:
IP addresses (192.168.1.1)
Ports (8080)
Connections between computers
Example:
One computer sends data
Another computer receives it
That communication is handled by net
difference btw net and http:
net	-- Low-level networking (connection, TCP, IP)
http -- High-level protocol built on top of net

ex : for creating web server
package main

import (
	"fmt"
	"net/http"
)

func handler(w http.ResponseWriter, r *http.Request) { 
	fmt.Fprint(w, "Web server created success")
}
// r *http.Request : contains incoming request w http.ResponseWriter Used to send response to client
func main() {
	http.HandleFunc("/", handler)
	err := http.ListenAndServe(":8080", nil)
	if err != nil {
		fmt.Println("Server is not running")
	}
}

//why here it uses *http.request: Because:
/*Request data can be large
Go does NOT copy it
It sends reference (address)*/

Routers:
A router decides: “Which handler should run for this URL.It maps routes to handlers.
Here the routing is handled by Go’s built-in net/http.
When Do We Use Routers?

Use routers when:
You have multiple APIs
You want clean code
You need dynamic URLs
You need middleware
You build real REST APIs
Example APIs:
GET    /students
POST   /students
PUT    /students/1
DELETE /students/1
Difference Between Handler and Router
Feature 	      	Handler				Router
Purpose 			Handles request		Selects handler
Role	 			Business logic		URL management
Works with			Request & Response	Paths & methods
Example				Login logic	/login → loginHandler
Go already has a basic router inside net/http.
